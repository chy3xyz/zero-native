## Defining a handler

```zig
fn ping(context: *anyopaque, invocation: zero_native.bridge.Invocation, output: []u8) anyerror![]const u8 {
    _ = invocation;
    const self: *App = @ptrCast(@alignCast(context));
    self.ping_count += 1;
    return std.fmt.bufPrint(output, "{{\"message\":\"pong\",\"count\":{d}}}", .{self.ping_count});
}
```

The handler writes its JSON result into the provided `output` buffer (max 12 KiB) and returns a slice of it. Results must be valid JSON values; invalid raw text is rejected with `handler_failed`. When returning user data as a string, use the bridge helper so quotes and control characters are escaped:

```zig
return zero_native.bridge.writeJsonStringValue(output, user_supplied_name);
```

## Wiring the dispatcher

```zig
fn bridge(self: *App) zero_native.BridgeDispatcher {
    self.handlers = .{.{ .name = "native.ping", .context = self, .invoke_fn = ping }};
    return .{
        .policy = .{ .enabled = true, .commands = &policies },
        .registry = .{ .handlers = &self.handlers },
    };
}
```

## Calling from JavaScript

```javascript
const result = await window.zero.invoke("native.ping", { source: "webview" });
console.log(result); // { message: "pong from Zig", count: 1 }
```

## Invocation

When a handler is called, it receives an `Invocation` with:

- `request.id` -- caller-provided request ID (max 64 bytes)
- `request.command` -- command name (max 128 bytes, no `/` or spaces)
- `request.payload` -- JSON payload string
- `source.origin` -- origin of the requesting page (e.g. `zero://app`)
- `source.window_id` -- which window sent the request

## Streaming channels

Channels stream typed values from Zig to JavaScript over the bridge using a lightweight wire protocol. Each channel is identified by a unique `StreamId` (a `u64`) and sends one or more **value frames** followed by an **end frame**.

### Frame format

Every frame is a tight JSON object with no extra whitespace:

| Field     | Type     | Description                          |
|-----------|----------|--------------------------------------|
| `stream`  | `u64`    | Channel identifier                   |
| `kind`    | `string` | `"value"` or `"end"`                |
| `payload` | JSON     | Serialized value or empty string     |

**Example value frame:**
```json
{"stream":1,"kind":"value","payload":42}
```

**Example end frame:**
```json
{"stream":1,"kind":"end","payload":""}
```

### Creating a typed channel

Use `bridge.Channel(T)` to send typed values. The writer callback delivers serialized frames through your bridge response path. Supported payload types: `i32`, `i64`, `u32`, `u64`, `f32`, `f64`, `bool`, and `[]const u8`.

```zig
const bridge = @import("zero-native").bridge;

// Test harness that collects frames into a list:
const TestSink = struct {
    frames: std.ArrayList([]u8),
    allocator: std.mem.Allocator,

    fn write(ctx: *anyopaque, frame_json: []const u8) anyerror!void {
        const self: *@This() = @ptrCast(@alignCast(ctx));
        const copy = try self.allocator.dupe(u8, frame_json);
        try self.frames.append(copy);
    }
};

var sink = TestSink{ ... };
var channel: bridge.Channel(i32) = .{
    .id = 1,
    .writer = TestSink.write,
    .writer_ctx = &sink,
    .allocator = std.testing.allocator,
};

// Send progress updates:
try channel.send(25);  // {"stream":1,"kind":"value","payload":25}
try channel.send(50);  // {"stream":1,"kind":"value","payload":50}
try channel.send(100); // {"stream":1,"kind":"value","payload":100}

// Signal completion (idempotent):
try channel.close();   // {"stream":1,"kind":"end","payload":""}
```

### Runtime integration

The runtime provides `createStreamChannel` as a convenience for wiring channels through the platform bridge. The function allocates a fresh `StreamId` and returns a `bridge.Channel(T)` whose writer automatically routes frames to `completeBridgeResponse` for the target window and webview. Until this helper is fully implemented, callers can construct a `bridge.Channel(T)` directly with a writer that calls their own bridge response path вЂ” the channel module tests demonstrate the full round-trip.

### Consuming a channel from JavaScript

Use the typed `window.zero.channels.open(id, handler)` API. The runtime sends value frames and a final end frame; the handler receives them through `onValue` and `onEnd`:

```typescript
const channel = window.zero.channels.open<number>(streamId, {
  onValue(value) { console.log(value); },
  onEnd() { console.log("done"); },
});
```
