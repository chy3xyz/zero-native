# Updates

zero-native reserves an explicit update configuration in `app.zon` so packaged apps can declare how they discover signed updates.

```zig
.updates = .{
    .feed_url = "https://example.com/releases/zero-native-feed.json",
    .public_key = "base64-ed25519-public-key",
    .check_on_start = true,
},
```

The runtime does not silently install updates. Apps should surface update checks through their own UI, verify signatures before applying artifacts, and keep platform-specific installation behavior explicit.

When `check_on_start` is `true` and the `updater` plugin is enabled, the runtime fetches the feed during plugin startup. Network failures on startup are ignored so the app still launches.

## Updater plugin

The `updater` extension plugin (`src/extensions/plugin_updater.zig`) implements the update lifecycle: check, download, and install.

### ModuleId

`108` — `{ .kind = .custom, .name = "updater" }`.

### API

**`create(allocator, io, current_version, manifest_url, public_key_b64)`**
Allocates updater state. `current_version` is a dotted version string (e.g. `"1.0.0"`). `manifest_url` is the update manifest endpoint. `public_key_b64` is an optional base64-encoded Ed25519 public key for signature verification (use `""` to skip).

**Commands**

| Command             | Description                                                       |
| ------------------- | ----------------------------------------------------------------- |
| `updater.check`    | Fetches manifest, compares versions, sets `update_available`.     |
| `updater.download` | Downloads the platform-specific update archive from the manifest. |
| `updater.install`  | Copies the downloaded archive into `$TMPDIR/zero-native-update/`. |

**State fields**

| Field              | Type                    | Description                                   |
| ------------------ | ----------------------- | --------------------------------------------- |
| `update_available` | `bool`                  | Set by `.check` when a newer version exists.  |
| `manifest`         | `?Manifest`             | Parsed manifest from the last `.check`.       |
| `downloaded`       | `bool`                  | Set by `.download` on success.                |
| `archive_path`     | `?[]u8`                 | Path to the downloaded archive file.          |
| `installed`        | `bool`                  | Set by `.install` on success.                 |
| `staging_path`     | `?[]u8`                 | Path to the staged update.                    |

The plugin does **not** restart the application. The host is responsible for swapping binaries on the next launch.

### Update manifest format

The manifest is a JSON document served at `manifest_url`:

```json
{
  "version": "1.0.0",
  "notes": "Optional release notes",
  "platforms": {
    "macos-aarch64": {
      "url": "https://example.com/zero-native-1.0.0-macos-aarch64.tar.gz",
      "signature": "base64-ed25519-signature",
      "size": 1048576
    },
    "linux-x86_64": {
      "url": "https://example.com/zero-native-1.0.0-linux-x86_64.tar.gz",
      "signature": "base64-ed25519-signature",
      "size": 983040
    }
  }
}
```

Platform keys follow the pattern `{os}-{arch}`: `macos-aarch64`, `macos-x86_64`, `linux-x86_64`, `linux-aarch64`, `windows-x86_64`.

### Key generation

Generate an Ed25519 keypair using `std.crypto`:

```zig
const std = @import("std");
const Ed25519 = std.crypto.sign.Ed25519;

const kp = Ed25519.KeyPair.generate(io);

// Print base64-encoded public key (publish this in app.zon).
var b64_buf: [64]u8 = undefined;
const pub_b64 = std.base64.standard.Encoder.encode(&b64_buf, &kp.public_key.bytes);
std.debug.print("{s}\n", .{pub_b64});

// Sign the update archive:
const archive = try std.fs.cwd().readFileAlloc(allocator, "update.tar.gz", 1 << 30);
defer allocator.free(archive);
const sig = try kp.sign(archive, null);
const sig_bytes = sig.toBytes();
const sig_b64 = std.base64.standard.Encoder.encode(&b64_buf, &sig_bytes);
std.debug.print("{s}\n", .{sig_b64});
```

### Integration steps

1. Generate a keypair and encode the public key as base64.
2. Add `public_key` and `feed_url` to `app.zon`.
3. Add `"updater"` to `.plugins` in `app.zon`. The runtime will configure the plugin using the `.updates` block automatically when you call `runtime.loadPlugins(..., metadata.plugins, .{ .config = ... })`.
4. Call `updater.check` (empty payload fetches the configured feed). Inspect `update_available` and surface the result in your UI.
5. On user confirmation, call `updater.download` then `updater.install`.
6. On next launch, the app host detects the staged update and swaps binaries.

### Network considerations

The updater plugin uses `http_client`, which supports both plain `http://` and `https://` URLs. HTTPS requests are handled by the httpz/OpenSSL backend; make sure the target system has `libssl`/`libcrypto` available when using HTTPS feeds.
